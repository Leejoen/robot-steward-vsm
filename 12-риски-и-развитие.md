# 12. Риски и развитие

## Риски

| Риск | Вероятность | Влияние | Митигация |
| --- | --- | --- | --- |
| Недоступность реальных данных расписания | Высокая | Среднее | Использовать тестовый адаптер и описать точку интеграции |
| Слишком широкий функциональный охват | Высокая | Высокое | Ограничить MVP программным прототипом сопровождения |
| Дублирование цифровой платформы вокзала | Средняя | Высокое | Зафиксировать робота как пользовательский канал и сценарный слой |
| Сложность физического робота | Высокая | Высокое | Вынести физическую платформу в развитие |
| Неполный контракт с цифровой платформой | Средняя | Высокое | Начать с mock-контракта и явно описать поля |
| Некорректный маршрут для маломобильного пассажира | Средняя | Высокое | Запрашивать безбарьерный режим и проверять предупреждения платформы |
| Непонятный интерфейс | Средняя | Среднее | Ручная UX-проверка на типовых сценариях |
| Избыточная архитектура для MVP | Средняя | Среднее | Сохранять модульность на уровне кода, но запускать локально |

## Развитие после MVP

- Подключение к реальной цифровой платформе умного вокзала.
- Интеграция с билетной системой или QR-сканером через платформу.
- Голосовой ввод и синтез речи.
- Мультиязычная поддержка.
- Подключение мониторинга пассажиропотоков.
- Компьютерное зрение для оценки загруженности.
- Интерфейс на физическом терминале или мобильной платформе.
- Интеграция с системой заявок персонала.
- Расширенная аналитика проблемных зон вокзала.

## Решения для пересмотра

| Решение | Когда пересмотреть |
| --- | --- |
| MVP как веб-прототип | Если появится требование к физическому демонстратору |
| SQLite для локального запуска | Когда появятся многопользовательские сценарии |
| Mock-адаптер цифровой платформы | При доступе к реальному API платформы |
| Локальная имитация маршрутов | При подключении к промышленному или пилотному расчету маршрутов |
| Ручной ввод текущей зоны | Если появится источник позиционирования пассажира или робота |

## Исследовательская оценка

Для оценки эффективности системы можно использовать:

- время построения маршрута;
- расчетное время прохода до цели;
- долю успешно построенных маршрутов;
- корректность безбарьерных маршрутов;
- устойчивость при закрытии зон;
- сравнение с маршрутом по статической схеме;
- количество обращений, обработанных без участия справочной службы;
- экспертную оценку понятности интерфейса.
