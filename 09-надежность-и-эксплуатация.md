# 09. Надежность и эксплуатация

## Цели надежности MVP

MVP должен быть устойчивым к ошибкам данных и понятным при сбоях. Для первой версии прототипа приоритетны корректная диагностика, безопасное поведение и прозрачное fallback-поведение.

## Типовые отказы

| Отказ | Поведение системы |
| --- | --- |
| Платформа недоступна | Показать ограниченный режим и предложить обратиться к персоналу или табло |
| Поезд не найден | Показать сообщение и предложить выбрать цель вручную |
| Зона не найдена в ответе платформы | Не показывать маршрут как корректный, зафиксировать ошибку адаптера |
| Маршрут невозможен | Объяснить причину и предложить обратиться к персоналу |
| Закрыта зона на маршруте | Перестроить маршрут без закрытого участка |
| Ошибка создания обращения | Показать, что заявка не создана, и предложить альтернативный канал |
| База данных недоступна | Перейти в состояние ошибки и записать диагностическое сообщение |

## Наблюдаемость

Минимальный набор сигналов:

- количество запросов сопровождения;
- количество успешных и неуспешных ответов платформы;
- среднее время ответа робота;
- количество запросов по поездке;
- количество обращений к персоналу;
- наиболее частые цели пассажиров;
- зоны, где чаще возникают проблемы;
- ошибки платформенного адаптера.

## Эксплуатационные сценарии

- Администратор меняет mock-ответы платформы.
- Администратор задает сценарий закрытого участка через mock.
- Администратор изменяет путь отправления тестового поезда через mock.
- Сотрудник просматривает обращения.
- Автор эксперимента выгружает статистику сценариев.

## Контроль качества данных

Перед демонстрацией требуется проверить:

- mock-ответы платформы покрывают основные сценарии;
- у тестовых поездов есть зона посадки или понятная причина отсутствия данных;
- безбарьерный сценарий содержит явный признак доступности;
- закрытая зона приводит к предупреждению или альтернативному маршруту;
- робот не скрывает отсутствие данных платформы.

## Recovery-подход

Для MVP достаточно ручного восстановления:

- перезапуск приложения;
- повторная загрузка mock-ответов платформы;
- очистка тестовых обращений;
- возврат сценариев сопровождения к базовым значениям.
