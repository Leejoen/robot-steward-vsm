# 00. Как ведется документация

## Цель

Этот раздел фиксирует правила ведения архитектурной документации первой итерации. Документы должны помогать проектировать систему, а не просто описывать идею в общих словах.

## Принципы

- Разделять принятые решения, допущения и открытые вопросы.
- Не расширять MVP за счет физического робота, компьютерного зрения и голосового интерфейса, пока не проверено ядро системы.
- Не дублировать цифровую платформу умного вокзала: карта, рейсы, расписание и базовые маршруты в целевой архитектуре приходят от платформы.
- Поддерживать единые названия компонентов: Robot Interaction UI, Steward Backend API, Dialogue/Scenario Service, Smart Station Platform Adapter, Assistance Service, Interaction Analytics.
- Связывать требования с архитектурой, данными, тестами и рисками.
- Для существенных решений добавлять ADR.
- Mermaid-диаграммы держать рядом с поясняющим текстом.

## Термины

| Термин | Значение |
| --- | --- |
| Робот-стюард | Целевой сервисный помощник пассажира на вокзале ВСМ |
| MVP | Программный прототип без физической робототехнической платформы |
| Сопровождение | Навигационная, информационная и сервисная помощь пассажиру |
| Цифровая платформа умного вокзала | Внешняя для робота система, которая хранит карту, рейсы, сессии и базовые подсказки |
| Тестовая модель вокзала | Локальный mock цифровой платформы для автономной демонстрации MVP робота |
| Обращение | Запрос пассажира к персоналу или сообщение о проблеме |

## Что считать готовностью первой итерации

- Есть описание системы и границ MVP.
- Есть требования и архитектурные драйверы.
- Есть C4-диаграммы, sequence/state/ER/deployment-диаграммы.
- Есть описание данных, развертывания, надежности, безопасности и тестирования.
- Есть минимум три ADR по ключевым решениям.
- Открытые вопросы не скрыты, а вынесены явно.
